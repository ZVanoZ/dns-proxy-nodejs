## Анализ существующих решений в сообществе

### Решение 1: Единый файл `todo-list.md` в корне проекта (РЕКОМЕНДУЕТСЯ) ⭐⭐⭐⭐⭐

**Описание:**
- Создается один файл `todo-list.md` в корне проекта или в директории `docs/`
- Все задачи перечисляются в этом файле с номерами и описанием
- Каждая задача может иметь ссылки на контекстные файлы
- Задачи берутся в работу по одной через команду `/van`

**Структура:**
```
project-root/
├── todo-list.md
├── memory-bank/
│   ├── tasks.md          # Текущая активная задача
│   ├── activeContext.md  # Текущий фокус
│   └── archive/          # Завершенные задачи
└── docs/
    └── prompt/           # Контекстные файлы для задач
```

**Пример `todo-list.md`:**
```markdown
Список задач, которые нужно добавить в план и контекст для них.

1. Используя методологию vanzan01/cursor-memory-bank v0.8 составить список задач.
   См. [geminy.md](001/geminy.md)

2. Улучшить и актуализировать документацию
   см. [question.md](002/geminy/question.md) и [answer.md](002/geminy/answer.md)

3. Актуализировать код и стек. см. [geminy.md](001/geminy.md)

4. Добавить поддержку протокола IPv6

5. Добавить сбор и журналирование статистики
```

**Плюсы:**
- ✅ Простота организации
- ✅ Легко добавлять новые задачи
- ✅ Все задачи в одном месте
- ✅ Не требует дополнительных инструментов
- ✅ Соответствует принципам методологии (одна активная задача в `memory-bank/tasks.md`)

**Минусы:**
- ❌ Нет встроенной поддержки статусов задач
- ❌ Нет автоматического отслеживания прогресса

**Оценка:** ⭐⭐⭐⭐⭐ (5/5) - **РЕКОМЕНДУЕТСЯ**

### Решение 2: Структурированные директории по задачам

**Описание:**
- Каждая задача имеет свою директорию с контекстными файлами
- Структура: `docs/prompt/<YYYY>/<YYYY-MM-DD>/<task-number>/`
- В каждой директории хранятся файлы контекста задачи

**Структура:**
```
docs/
└── prompt/
    └── 2026/
        └── 2026-01-21/
            └── 001/
                ├── task.md
                ├── geminy.md
                └── todo-list.md  # Ссылки на задачи
```

**Плюсы:**
- ✅ Хорошая организация контекстных файлов
- ✅ Легко найти все файлы, связанные с задачей
- ✅ Подходит для больших проектов

**Минусы:**
- ❌ Более сложная структура
- ❌ Может быть избыточно для простых задач
- ❌ Требует больше места

**Оценка:** ⭐⭐⭐⭐ (4/5) - Подходит для больших проектов

## Рекомендуемая структура для проекта dns-proxy-nodejs

### Вариант A: Единый файл `todo-list.md` (РЕКОМЕНДУЕТСЯ) ⭐⭐⭐⭐⭐

**Структура:**
```
project-root/
├── todo-list.md                    # Список всех задач
├── memory-bank/
│   ├── tasks.md                    # Текущая активная задача (одна)
│   ├── activeContext.md            # Текущий фокус
│   ├── archive/                    # Завершенные задачи
│   │   └── archive-[task_id].md
│   ├── creative/                   # Творческие фазы
│   │   └── creative-[feature].md
│   └── reflection/                 # Рефлексии
│       └── reflection-[task_id].md
└── docs/
    └── prompt/                     # Контекстные файлы
        └── 2026/
            └── 2026-01-21/
                └── 001/
                    ├── task.md
                    ├── geminy.md
                    └── reference/   # Документация
```

**Принципы:**
1. **Одна активная задача:** В `memory-bank/tasks.md` всегда только одна текущая задача
2. **Список задач:** Все задачи перечислены в `todo-list.md`
3. **Контекст:** Контекстные файлы хранятся в `docs/prompt/`
4. **Архив:** Завершенные задачи архивируются в `memory-bank/archive/`

**Workflow:**
1. Пользователь добавляет задачи в `todo-list.md`
2. Выбирает задачу из списка
3. Выполняет `/van` с описанием задачи
4. Работает над задачей через полный цикл методологии
5. После `/archive` переходит к следующей задаче из списка
