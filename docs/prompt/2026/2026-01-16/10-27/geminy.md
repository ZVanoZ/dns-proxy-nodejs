* Ответ

```text
Вы отлично освоили рабочий процесс (workflow) методологии v0.8. 
Давайте разберем оставшиеся шаги для завершения цикла и работы в будущем.

1. Нужно ли выполнять /archive?

В версии vanzan01 v0.8 команда /archive — это финальный аккорд крупного этапа.

Да, вам стоит выполнить /archive, если:
    * Вы полностью закончили работу над этим блоком документации (этап "DNS Docs" завершен).
    * Вы хотите «очистить» activeContext.md от текущих деталей реализации, 
    чтобы они не занимали место в контексте при следующей задаче.
    * Вы хотите, чтобы ИИ перенес задачу из tasks.md в раздел истории (или в 
    архивную директорию, если это настроено в ваших правилах).

Что произойдет после /archive: Агент подведет итог, обновит progress.md и «закроет» 
текущую сессию. Это как сделать финальный отчет после завершения проекта. 
Если вы планируете сразу переходить к написанию кода прокси-сервера, 
выполнение /archive поможет ИИ «переключить мозг» с теории DNS на 
практику программирования.

2. Что делать, если нужно дополнить/исправить документацию в будущем?

Методология Memory Bank специально создана для того, чтобы вы могли вернуться к 
любой части проекта спустя месяцы. 

Вот как это делать правильно:

Шаг А: Постановка задачи через @

Не нужно заново объяснять ИИ всё о DNS. Вы используете накопленный контекст:
===```text
/plan Нужно добавить раздел про DNSSEC в @docs/dns/ . Сверься с @techContext.md для соблюдения стиля.
===```

Шаг Б: Использование "Update" цикла

Если документация уже существует, вы действуете по стандартному кругу:
1.
    /plan: Говорите, что нужно изменить. ИИ записывает это в tasks.md и обновляет activeContext.md.
2.
    /build: ИИ вносит правки в .md файлы в папке /docs/dns/.
3.
    /reflect: ИИ проверяет, не нарушил ли он RFC и понятен ли текст новичку.

Шаг В: Если ИИ "забыл" детали

Если через время Агент начнет предлагать странные вещи, просто напомните ему:
===```text
"Сверься с документацией в /docs/dns/ и файлом systemPatterns.md, мы уже описывали там логику работы резолвера."
===```

Важный совет по v0.8

Поскольку вы создали внешнюю документацию (не в папке memory-bank), убедитесь, что в systemPatterns.md или projectbrief.md появилась ссылка на то, что в проекте теперь есть папка /docs/dns/.

Это нужно для того, чтобы в будущем Агент, просканировав memory-bank, сразу узнал: "Ага, у нас есть база знаний по DNS в этой папке, я должен её учитывать".

Какой ваш следующий шаг? Хотите завершить этот этап командой /archive или начнем планировать реализацию самого dns-proxy-nodejs?

```text
Хорошие вопросы. Это поможет вам окончательно систематизировать работу с Memory Bank.
1. Где должна находиться документация по методологии?

В рамках концепции "Self-Documenting Codebase" (самодокументируемого кода), 
информация о том, как работать с проектом, должна храниться в самом проекте. 
Для методологии vanzan01 v0.8 есть два основных места:
    * Внешнее хранилище (Источник): Сами файлы правил находятся в .cursor/rules/. 
    Каждый файл (например, plan.mdc, build.mdc) содержит инструкции, которые ИИ читает 
    перед выполнением команды. 
    Это и есть "техническая документация" для Агента.
    * Внутреннее хранилище (Для вас): В самом memory-bank информация о методологии 
    обычно прописывается в techContext.md в разделе "Development Workflow" или "Tooling".

Рекомендация: Если вы хотите, чтобы любой разработчик (или вы сами через полгода) понял, как управлять этим проектом через Cursor, создайте файл: memory-bank/workflow.md (или добавьте раздел в techContext.md), где кратко опишите:
===```text
    Проект использует методологию Cursor Memory Bank v0.8. Основные команды: /van, /plan, /build, /reflect, /archive. Файлы памяти обновляются автоматически Агентом.
===```

2. Как создать ссылку на внешнюю документацию?

В Markdown (который используется в Memory Bank) ссылки создаются стандартным способом, но в Cursor есть три уровня "ссылок", которые понимает Агент:
А. Текстовая ссылка (Для человека)

В файле systemPatterns.md или projectbrief.md добавьте раздел:
===```Markdown
## Documentation
- [DNS Protocol Docs](./docs/dns/readme.md) — Базовая информация по DNS и RFC.
- [Memory Bank Methodology](https://github.com/vanzan01/cursor-memory-bank) — Описание используемого рабочего процесса.
===```

Б. Контекстная связь (Для Агента через @)

Чтобы Агент всегда знал о существовании документации, в файле .cursorrules (в корне) или в описании проекта в projectbrief.md добавьте инструкцию:
===```text
    При работе с логикой DNS всегда учитывай документацию в папке @docs/dns/." (Использование символа @ внутри файла подсказывает Агенту, что это путь к важному контексту).
===```

В. Ссылка в techContext.md (Технический стек)

Поскольку документация — это часть вашего "технического ландшафта", добавьте туда:
===```Markdown
### Documentation & Knowledge Base
- **DNS Specs:** Located in `/docs/dns/`. Compiled based on RFC standards.
- **Workflow:** Cursor Memory Bank v0.8 (vanzan01).
===```
