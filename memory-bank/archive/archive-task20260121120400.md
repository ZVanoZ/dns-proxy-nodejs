## IMPLEMENTATION

### Конфигурация и Options

- INI-конфиг (`config/app.ini.dist`, `env/dev/prod/config/app.ini.dist`, `config/app.ini`):
  - `[app]` с полями:
    - `v4host` (опционально),
    - `v6host` (опционально),
    - `port` (общий порт для обоих сокетов).
  - Правило: хотя бы один из `v4host`, `v6host` должен быть задан.
- Options (`src/App/Options/index.ts`, `OptionsLoader.ts`):
  - `bindV4Host?: string`, `bindV6Host?: string`, `bindPort: number`.
  - Загрузка `v4host`, `v6host`, `port` из INI, валидация на «хотя бы один хост».

### Класс App (dual-stack)

- Два сокета:
  - `v4Socket: Socket | null` (udp4),
  - `v6Socket: Socket | null` (udp6).
- `initSocket()`:
  - создаёт udp6 при наличии `bindV6Host` (с `reuseAddr: true`, `ipv6Only: true`), затем в callback успешного bind создаёт udp4 при наличии `bindV4Host` (с `reuseAddr: true`);
  - таким образом устраняется конфликт EADDRINUSE при dual-stack на одном порту в Docker/Linux.
- `onSocketMessage(msg, rinfo, socket)`:
  - единый обработчик для обоих сокетов, третий аргумент — конкретный сокет, с которого пришёл запрос;
  - вызывает `getIp(dnsName)` и далее `sendSuccessResponse`/`sendErrorResponse` с тем же `socket`.

### Кеш и answer-source

- Структура кеша (DnsCache) и `answer-source` (предыдущая задача) переиспользуются, изменений под IPv6 не потребовалось:
  - кеш ключуется по DNS-имени, не по адресу клиента;
  - источник ответа (`answerSource`) логируется в `sendSuccessResponse`.

### Тесты

- Обновлён `config/test.config.ini` под новую конфигурацию.
- Обновлены/добавлены тесты:
  - запросы по IPv4 (основной сценарий);
  - опционально запросы по IPv6 при наличии тестовой конфигурации;
  - всего: 31–32 теста, все проходят.

### Docker и окружение

- `docker-compose.yml`:
  - сеть `dns-proxy-nodejs-net` в режиме dual-stack:
    - `enable_ipv6: true`,
    - IPv4-подсеть `${NETWORK_SUBNET}`,
    - IPv6-подсеть `${NETWORK_SUBNET_IPV6:-fd00:0:0:1::/80}`.
  - сервисы:
    - `dns-proxy-dev`: `ipv4_address: ${DEV_SERVER_IP}`, `ipv6_address: ${DEV_SERVER_IPV6:-fd00:0:0:1::133}`;
    - `dns-proxy-prod`: `ipv4_address: ${PROD_SERVER_IP}`, `ipv6_address: ${PROD_SERVER_IPV6:-fd00:0:0:1::13}`.
- `.env` / `.env.dist`:
  - IPv4: `DEV_SERVER_IP`, `PROD_SERVER_IP`, `NETWORK_SUBNET`;
  - IPv6: `DEV_SERVER_IPV6`, `PROD_SERVER_IPV6`, `NETWORK_SUBNET_IPV6` (в .env.dist — как комментарии/дефолты для compose).

### Дополнение № 1: Логирование интерфейса и questionName

- Типы (`src/App/types.ts`):
  - `IpVersion = 'IPv4' | 'IPv6'`,
  - `ServerInterfaceInfo { ip: string; port: number; family: IpVersion }`.
- Helper (`src/App/index.ts`):
  - `getServerInterfaceInfo(socket: Socket): ServerInterfaceInfo` использует `socket.address()` для получения локального IP, порта и семейства.
- `sendErrorResponse`:
  - безопасно извлекает `questionName` из `dnsRequest.questions[0].name`, если запрос декодирован;
  - логирует `questionName` и `serverInterface` при начале и при отправке ответа.
- `sendSuccessResponse`:
  - добавляет `serverInterface` в структурный лог вместе с `answer-source` и `responseAnswers`.

## LESSONS LEARNED

- Для UDP dual-stack на одном порту в Linux критичны:
  - порядок bind (сначала IPv6 с `ipv6Only: true`, затем IPv4),
  - настройки `reuseAddr`, `ipv6Only`.
- План/CREATIVE фаза для задач уровня 3 даёт устойчивый каркас, минимизируя переделки на стадии BUILD.
- Чеклисты BUILD должны явно включать Docker и `.env`, чтобы сетевые и окруженческие изменения не терялись.
- При доработках логирования важно сохранять структурированный формат и расширять его эволюционно (добавлением полей).
- Optional chaining и строгая типизация (`ServerInterfaceInfo`, `IpVersion`) упрощают безопасный доступ к данным и уменьшают количество ошибок.
